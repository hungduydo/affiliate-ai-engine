## P2 — Post-discovery-launch

### [DISCO-2] Phase 2: Multi-network discovery (CJ + Shopee aggregated)

**What:** A single `/products/discover` page that queries both CJ and Shopee affiliate APIs,
aggregates results, and ranks by a cross-network EPC-equivalent signal.

**Why:** CJ alone limits the opportunity surface. Some high-margin products are Shopee-only.
Shopee is the primary affiliate network in Vietnam/SEA.

**Pros:** Doubles the product surface for discovery.

**Cons:** Shopee API is fragile (Playwright scraping, brittle). Requires cross-network score
normalization. The Shopee affiliate API is poorly documented.

**Context:** Deferred from discovery v1 (`/office-hours` session 2026-04-04, Approach B from
alternatives considered). Build after discovery v1 ships and provides real usage data.
The existing `ShopeePlaywrightAdapter` in `source-connector/infrastructure/adapters/shopee/`
provides a starting point, but the adapter is brittle.

**Effort:** Human ~2 months / CC+gstack ~1 week
**Priority:** P2
**Depends on:** Discovery v1 shipped first. Real usage data confirming discovery feature adds value.

### [DISCO-3] Per-user cache isolation for SaaS launch

**What:** The current `DiscoveryCache` uses a global `"cj_all"` cache key. When multi-tenancy
ships, each user needs their own cache key, scoped to their CJ credentials.

**Why:** Without this, User A's cached discovery results are served to User B (wrong data if
they have different CJ accounts).

**Pros:** Required for correct multi-tenant behavior.
**Cons:** Small change, easy to defer while single-user.

**Context:** Flagged during CEO review on 2026-04-04. The fix is a one-line key change:
`"cj_all"` → `"cj_${userId}"`. But it requires the auth module to be complete so `userId`
is available at request time.

**Effort:** Human ~2h / CC+gstack ~5min
**Priority:** P2 (gate: multi-tenancy / auth module ships)
**Depends on:** Auth module (started in codebase, not yet complete)

## P3 — Structural clean-up

### [DISCO-4] Move DiscoveryCache from config_db to a more appropriate home

**What:** `DiscoveryCache` is currently in `config_db` because it reuses the existing Prisma
config client. Semantically, discovery data belongs closer to products_db.

**Why:** `config_db` is for prompt templates and connector config — not discovery results.
As discovery grows (per-user, multi-network), the config_db home will cause confusion.

**Pros:** Cleaner module boundaries. Easier future extension.
**Cons:** Requires a migration + creates a new challenge (source-connector has no DB, so it
would either need a new DB or the cache must live in products_db with access via internal REST).

**Context:** Flagged during CEO review on 2026-04-04. Short-term config_db placement is a
pragmatic choice (reuses existing Prisma client, avoids new infrastructure). But it's
acknowledged debt.

**Effort:** Human ~4h / CC+gstack ~15min
**Priority:** P3 (low urgency — purely structural)
**Depends on:** None

### [DISCO-5] E2E test for import-discover flow

**What:** End-to-end test covering: click "Import & Generate" on ProductDiscoverPage → modal opens pre-filled from localStorage → confirm → POST /source-connector/ingest-discover → navigate to /content/:id (ContentEditorPage).

**Why:** This cross-component flow spans ProductDiscoverPage, the import modal, two backend API calls (products + content), and a frontend navigation. Unit tests can't verify the full journey. If the navigation target changes or the modal state breaks, nothing catches it.

**Pros:** High confidence in the most user-visible happy path. Catches regressions from future modal changes.
**Cons:** Requires E2E test setup (Playwright or similar). More brittle than unit tests.

**Context:** Identified during eng review 2026-04-05. The ingest-discover handler makes 2 internal HTTP calls sequentially — the E2E test is the only way to verify the combined response shape and navigation target are correct together.

**Effort:** Human ~4h / CC+gstack ~15min
**Priority:** P2 (add after feature ships)
**Depends on:** Discovery v1 shipped

### [DISCO-6] DiscoveryCache Phase 2 composite key

**What:** Current `DiscoveryCache` Prisma model uses a fixed key `"cj_all"`. Phase 2 multi-network support requires a composite key (e.g., `network + userId` or `network + key`).

**Why:** The single-row design is a Phase 1 constraint. When Phase 2 adds Shopee or per-user caching, the schema needs a composite primary key and the cache read/write logic needs to handle multiple rows.

**Pros:** Unlocks multi-network and multi-tenant caching cleanly.
**Cons:** Schema migration required. PUT/GET cache internal endpoints need updated contracts.

**Context:** Explicit Phase 1 constraint documented in plan (2026-04-04). Schema comment will say "Phase 1: single row. Phase 2 composite key needed." This TODO captures the intent so it's not forgotten when Phase 2 starts.

**Effort:** Human ~2h / CC+gstack ~10min
**Priority:** P2 (gate: Phase 2 multi-network or multi-tenant ships)
**Depends on:** [DISCO-2] Phase 2 multi-network discovery

## Pre-conditions (must verify before coding discovery feature)

These are BLOCKING. Verify before writing `ProductDiscoveryService`.

1. **CJ Product Search `advertiser-id` param**: Call `GET https://product-search.api.cj.com/v2/product-search?advertiser-id=XXXX&keywords=test` with a known advertiser ID. Confirm the param is accepted and filters results to that advertiser. If not supported, fall back to keyword search with advertiser name.

2. **CJ Advertiser API EPC fields**: Call `GET https://advertiser.api.cj.com/v2/advertiser-lookup?requester-cid={websiteId}&advertiser-ids=joined`. Confirm the response includes per-advertiser EPC fields (likely `network-rank`, `seven-day-epc`, `three-month-epc`). If EPC is not returned for your CJ account tier, the scoring model must be redesigned (commission-only ranking, or use a different data source).

These are the highest-risk unknowns in the plan. Verify before writing code.
